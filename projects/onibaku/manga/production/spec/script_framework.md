# 漫画脚本フレームワーク

この文書は、小説を漫画化するときに `storyboard.json` へ直接圧縮しないための中間レイヤーを定義する。

脚本作成前に `script_craft_research.md` も読むこと。そこには漫画原作、セリフ、会話劇、ネーム前脚本の調査メモをまとめている。

## 前提

今回の初回生成では、小説の内面独白を短いモノローグと絵指定へ圧縮したため、漫画として必要な会話の往復、沈黙、反応、距離感の変化が不足した。

性別設定の誤りは表層の問題であり、根本原因は **小説からネームへ進む前に、漫画脚本として場面を再構成する工程がなかったこと** にある。

## レイヤーの責務

| レイヤー | 目的 | 決めること | 決めないこと |
|---|---|---|---|
| 小説 | 内面・文体・時間圧縮で読ませる | 出来事、心理、語りの温度 | コマ割り、吹き出し量 |
| 翻案設計 | 何を漫画化するか選ぶ | 話数配分、変化量、ヒキ | 個別セリフの完成形 |
| 漫画脚本 | 会話劇として場面を成立させる | セリフ、反応、沈黙、サブテキスト、場面目的 | 詳細なコマ割り、構図 |
| ネーム | 読者体験へ変換する | ページ配分、コマ割り、めくり、視線誘導、吹き出し位置 | 物語の根本方針 |
| `storyboard.json` | 画像生成へ渡す | ページ・コマ・絵指定・正確な文字列 | 脚本上の試行錯誤 |

## 漫画脚本の役割

漫画脚本は、ネームの上位概念として次を固定する。

- その場面で誰が何を欲しがっているか
- セリフの表面意味と裏の意味
- セリフ後に相手がどう反応するか
- 沈黙、目線、手元、距離、姿勢で何を語るか
- どの情報を言わせず、絵で出すか
- 1ページごとに読者を次へ進ませる疑問

漫画脚本は小説の要約ではない。小説の内面を、会話・行動・反応に翻訳するための制作文書である。

## 基本フォーマット

各話の脚本は `manga/epXX-*/production/spec/script.md` に置く。

```md
# 第X話「タイトル」漫画脚本

## episode_premise

この話で読者に発生させる問いを書く。

## emotional_delta

- 開始: 主人公は何を信じているか
- 終了: 主人公は何を見ないことにしたか

## must_say

- 必ず入れるセリフ

## must_not_say

- 説明しすぎになるため禁止するセリフ

## visual_motifs

| モチーフ | 初出 | 変化 | 意味 |
|---|---|---|---|

## scene_blocks

### S1 場所 / 時間

- 目的:
- 登場人物:
- 場面の入口:
- 場面の出口:
- 読者に残す疑問:

#### dialogue_ladder

| 段 | 発話者 | セリフ | 表面意味 | 裏の意味 | 相手の反応 |
|---|---|---|---|---|---|
| 1 |  |  |  |  |  |

#### reaction_beats

- セリフではなく、目線・手・姿勢・距離で見せる反応。

#### page_intent

| 想定P | 読者に残す感情 | ヒキ/メクリ | ネームへの注意 |
|---|---|---|---|
```

## 会話劇寄りにするための規則

1. 重要場面は「説明」ではなく「応酬」で書く。
2. 1つの決断には、最低でも `提案 -> 拒否/回避 -> 押し返し -> 沈黙 -> 受諾/保留` の段階を作る。
3. セリフを増やすときも、説明セリフではなく、相手を動かすセリフにする。
4. 各セリフには相手の反応を付ける。反応がないセリフは、漫画では空中に浮く。
5. モノローグは事実説明ではなく、自己欺瞞・言い訳・気づかないふりに使う。
6. 沈黙は「無」ではなく、目線、手元、距離、道具で書く。
7. 男同士の距離感、気まずさ、上下関係の曖昧さを、身体距離と敬語の揺れで出す。

## ネームへ渡す条件

漫画脚本からネームへ進める条件は次の通り。

- **`script_review.md` の判定が ship であること**（レビュー未実施では進めない）。
- 各シーンに入口と出口がある。
- 主要な決断が会話の段階を経ている。
- セリフ直後の反応が書かれている。
- ページごとの読者感情が仮置きされている。
- モチーフがどの場面でどう変化するか分かる。
- `must_not_say` によって説明過多を抑制できている。

## storyboard へ渡す条件

ネームから `storyboard.json` へ進める条件:

- **`name_review.md` の判定が ship であること**（レビュー未実施では進めない）。
- 脚本の `must_say` がネーム上で落ちていない。
- caption / monologue の分類が確定している。

## レビュー基準

脚本レビューでは、生成前に次を判定する。

| 観点 | 合格条件 | 失敗例 |
|---|---|---|
| 会話量 | 重要場面に応酬がある | 1つの発話で決断が終わる |
| 反応 | セリフ後の目線・姿勢・沈黙がある | セリフだけが並ぶ |
| サブテキスト | 表面意味と裏の意味が分かれている | 本音を説明してしまう |
| 漫画化可能性 | 絵にできる行動へ分解されている | 抽象心理だけで進む |
| ページ推進力 | 各ページに次へ進む問いがある | ただ状況説明が続く |
| モノローグ | 原則不使用。使う場合は同一コマの未整理断片のみ | 過去形・結果・俯瞰を monologue にしている |

## `storyboard.json` との関係

`storyboard.json` は脚本の代替ではない。脚本とネームで決まった内容を、画像生成に必要な粒度へ変換した実行仕様である。

したがって、今後は次の順序を守る。

1. `script.md` を作る。
2. `script_review.md` で脚本レビュー（**ship までネーム禁止**）。
3. `name_v2.md` でネーム相当のページ配分を決める。
4. `name_review.md` でネームレビュー（**ship まで storyboard 禁止**）。
5. `storyboard.json` に落とす。
6. `lint` / `review` / `prompts` を通してから画像生成する。
