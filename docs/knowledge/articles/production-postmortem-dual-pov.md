# 制作ポストモーテム: デュアルPOVリプレイ構成

16P読み切り。リプレイ構成（同じ月曜日を彼女視点8P→彼視点7P→同時通知→翌朝）。

## 構成設計で効いたもの

- **リプレイ構成**: 前半の「失恋に見える一日」が後半で全シーン意味反転する。通知の驚きを捨てて
  「読者だけが知っている」期待に変換。クラフト知見でいう「変化量」を構成自体で稼ぐ
- **対称構造**: 両陣営の友人が同じ語（「保険」）でアプリを紹介。セリフ反復のテーマ刷り込み（松井理論）と同型
- **告白を描かない**: 通知「両思いの人がいます」を物語のクライマックスに置き、トレーラーとして
  「この通知の先はあなたの物語」で終える。プロダクトの瞬間=感情のピークに一致させる
- **制服一貫性のための舞台制約**: 夜シーンを帰り道・部屋着指定込みのキャラ定義で処理。
  衣装バリエーションはキャラ定義に明記しないと崩れる

## ハーネス面で起きたこと

1. **ビジョンQAの初回較正が過剰**: 16ページ中11をfail判定。原因は「スロット不一致」と「読み順破壊」の混同
   ＋左右認識の自己矛盾（配置を正しく記述しながらswappedと結論）
2. **対策**: QA手順を「グリッド転記→読者順を歩いてreader_sequenceを出す→ascendingならorder_ok」の
   2段に分離。スロット逸脱はwarn。再判定で13 pass / 3 warn / 0 fail となり、人間の目視判断と一致
3. **QA出力のトークン切れ**: gridをJSON配列で冗長に書いてunparseable。「1行の文字列で簡潔に」指示＋
   max_tokens 4000で解消
4. **編集レビュー（review）が論理穴を検出**: 「P14で主人公がフルネームを入力できる根拠が作中にない」。
   lint（構文）にもビジョンQA（画像）にも原理的に見えない欠陥で、検出階層 lint=構文 / review=物語 /
   qa=実画像 の分担が確立した

## 教訓

- LLM評価器は「判定手順を構造化」しないと結論で自滅する。観察と判定を分けさせる
- 無言・モノローグ主体のムードページは内容マッチが曖昧になり、QAが甘くなる（既知の限界）
- ご都合主義チェック（裏付けの有無）は生成前のreviewに置くのが最安。画像になってからでは1ページ$0.18の修正になる
