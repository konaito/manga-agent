# 制作ポストモーテム: 読み順逆転と max_tokens

16P読み切り。文化祭デュアル構成版。

## 起きたこと

1. **max_tokens=1024で2コール分の課金だけ発生**。gpt-5.4-image-2は1ページ約5,700completion tokens（image_tokens約4,700含む）を使うため、1024では`finish_reason: length`で画像が返らない。**それでも$0.18課金される**
2. **2コマ横並びの読み順逆転が2ページで発生**（P2: 答え→質問の順、P10: 同様）。散文のレイアウト指示だけでは、モデルは右左を約2-3割の確率で入れ替える
3. P4・P7でもコマ順の入れ替えがあったが、偶然「読める」順序だったため採用

## 原因

- 画像トークンを考慮しないデフォルト値
- レイアウト指示が「5 panels in a clear Japanese manga layout...」という散文で、各コマの位置が非決定だった

## 対策（実装済み）

- max_tokensデフォルト16384（mangagen）
- **質問コマをワイド（全幅）にする**と物理的に逆転できなくなる。これが一番確実
- 後にper-panel `pos` スロット＋lintの横並びQ→A検出として機械化

## 教訓

- 失敗した生成にも課金される。事前検証（lint・dry-run）を惜しまない
- 「読める逆順」は採用してよい場合がある（伏せ→画面インサート等の漫画的イディオムとして成立する）。
  機械判定をfailにせずwarnに留め、最終判断は人間がする設計が正しい
